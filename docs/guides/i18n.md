# Internationalization

Agent UI ships built-in UI dictionaries for `en`, `ja`, `ko`, `zh-CN`, `es`,
and `fr`. Hosts own the active locale, the same way they own theme state.

```tsx
import {
  AgentChat,
  AgentLocaleSelect,
  AgentProvider,
  type AgentLocale,
} from "@nyosegawa/agent-ui-react";
import { useState } from "react";

function App({ transport }: { transport: AgentTransport }) {
  const [locale, setLocale] = useState<AgentLocale>("ja");

  return (
    <AgentProvider transport={transport}>
      <AgentLocaleSelect value={locale} onChange={setLocale} />
      <AgentChat
        locale={locale}
        messages={{
          "composer.placeholder": "フォローアップの変更を求める",
        }}
      />
    </AgentProvider>
  );
}
```

`AgentChat.locale` accepts any string and normalizes it to the supported locale
set, so browser values like `ja-JP`, `ko-KR`, and `zh-Hans-CN` are safe. Omit
the prop to infer from `navigator.language`; server-side and non-browser
renders fall back to English.

`messages` is a partial override map. Agent UI merges overrides over the
built-in dictionary, so hosts only provide product-specific copy. Use overrides
for wording that depends on the host workflow, for example changing `Send` to
`Run` or replacing the composer placeholder.

Agent UI only localizes library UI chrome: buttons, placeholders, labels,
menus, empty states, approval copy, status chrome, and accessibility labels.
It does not translate transcript messages, command output, tool output, model
responses, file paths, diagnostics returned by a bridge, or host-rendered slot
content.

For primitive-only layouts, wrap the rendered surface in `AgentI18nProvider`.
This is useful when the host uses `AgentStatusBar`, `AgentThreadView`, or
`AgentComposerPanel` without `AgentChat`.

```tsx
<AgentI18nProvider locale={locale} messages={messages}>
  <AgentThreadView />
</AgentI18nProvider>
```
